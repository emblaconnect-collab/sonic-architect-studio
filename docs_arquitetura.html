Ecossistema de Recepção - Estúdios

Documentação Técnica de Arquitetura & Implementação (Solo CEO)

Meta: Substituir plataformas engessadas (como Quaddro) por uma arquitetura nativa focada em conversão de alto ticket, usando o Google Agenda como "Banco de Dados" e a InfinitePay como gateway primário.

Stack Tecnológica: Next.js (Front-end e Server Routes) | Google OAuth (Login Clientless) | n8n (Back-office automation) | InfinitePay (Checkout & PIX)

Fase 1: Preparação do Terreno (APIs Google)

Passo 1.1

Configurar Projeto no Google Cloud Console

  1. Acesse o Google Cloud Console.
  2. Crie um novo projeto (Ex: "Pinaculo-Agenda-App").
  3. No menu "APIs e Serviços" > "Biblioteca", ative as seguintes APIs:
    • Google Calendar API (Para agendar e verificar horários)
    • Google Places API (New) (Para buscar os reviews 5 estrelas do estúdio)
Passo 1.2

Configurar o OAuth 2.0 (Login do Cliente)

  1. Vá em "Tela de Consentimento OAuth". Selecione "Externo".
  2. Preencha os dados básicos (Nome do App: Reservas do Estúdio). Só marque a obrigatoriedade dos escopos básicos (email, profile).
  3. Vá em "Credenciais" > Criar Credenciais > "ID do cliente OAuth".
  4. Selecione "Aplicativo Web".
  5. Adicione as Origens JavaScript autorizadas (ex: http://localhost:3000 para dev e https://www.site-do-cliente.com.br em prod).
  6. Adicione as URIs de redirecionamento autorizadas (A rota no Next.js que receberá o callback do Google, ex: /api/auth/callback/google usando NextAuth).
  7. Guarde o Client ID e o Client Secret no seu .env.local.
Passo 1.3

Credenciais de Serviço (O Robô da Agenda)

  1. Em "Credenciais", crie também uma "Conta de Serviço" (Service Account).
  2. Gere uma chave JSON para esta conta. Isso será usado pelo n8n para conseguir editar a agenda do dono do estúdio sem precisar de login.
  3. Ação no Estúdio: O dono do estúdio (ex: Decio) precisa entrar no Google Agenda dele, ir nas Configurações da Agenda principal, e "Compartilhar com pessoas específicas". Ele adiciona o e-mail da Conta de Serviço (ex: robo-agenda@pinaculo-app.iam.gserviceaccount.com) dando permissão de "Fazer alterações nos eventos".

Fase 2: O Front-End e Lógica no Next.js (Projeto `studio-site`)

Passo 2.1

Integração do NextAuth.js

  1. No seu projeto Next.js, instale: npm install next-auth.
  2. Configure a rota de API em app/api/auth/[...nextauth]/route.ts usando o GoogleProvider com o Client ID e Secret do Passo 1.2.
  3. Isso garante que ao clicar "Agendar", o modal OAuth abre rápido e captura o Gmail limpo.
Passo 2.2

Fluxo de Reserva na Interface

  1. Tela Inicial: Consumir a rota do n8n que lista "Horários Livres".
  2. Seleção: Cliente escolhe Dia/Hora.
  3. O Login Rápido: Dispara signIn('google'). O usuário é logado e temos o session.user.email na mão.
  4. Finalização: Interface chama a API Route interna do Next (ex: /api/checkout/pix).
Passo 2.3

A API Route de Geração de PIX (Server-side)

Atenção LGPD/Segurança: Toda requisição à InfinitePay, portando os Tokens deles, MUST ser feita num componente Backend. NUNCA no "use client".
  1. Na rota /api/checkout/pix, você recebe os dados do Frontend (Data, Hora Escolhida, Email capturado via sessão).
  2. Faça um POST direto do Next.js para a API da InfinitePay enviando o valor do ensaio.

    Detalhe Crucial: Configure o payload da InfinitePay definindo a URL do seu Webhook do n8n no campo designado (ver documentação IP).
  3. Devolva a imagem em base64 do QR Code para o Frontend exibir.

Fase 2.5: Credenciais e Setup na InfinitePay

Passo 2.5.1

Obtenção da API Key e Configuração do Webhook

  1. Cadastro Dev: Acesse o painel da sua conta InfinitePay pelo navegador e vá na seção de "Desenvolvedores" ou "Integrações" (API).
  2. Gerar API Key: Crie um novo Token de Acesso (API Key) para o ambiente de Produção. Guarde essa chave, ela nunca mais será exibida na tela. Cole-a imediatamente na sua Vercel / .env.local como INFINITEPAY_API_KEY.
  3. Validando os Endpoints de Pagamento: A cobrança via PIX usará um Endpoint de POST (geralmente /v2/orders ou /v2/pix conforme documentação v2 deles). O Authorization header deve levar esse Bearer Token.
  4. O Webhook Oficial: Note que o campo webhook_url vai no corpo da sua requisição quando for criar o pedido na API deles. É esse campo que vai dizer para a InfinitePay bater na porta do seu n8n.

Fase 3: A Engenharia Invisível (n8n)

O n8n será o seu cérebro. Aqui só existem Webhooks conversando com APIs.

Passo 3.1

Workflow de Confirmação e Fechamento de Agenda

  1. Nó Principal (Webhook Trigger): Crie um nó configurado para receber POST. Essa é a URL que você cadastrou lá na geração do PIX na InfinitePay. A InfinitePay manda JSON aqui.
  2. Nó IF (Filtro de Status): Crie um nó "IF" que valida se json.status == "approved" ou "paid".
  3. Nó Google Calendar (Create Event):
    • Autentique usando a Conta de Serviço (arquivo JSON) do Passo 1.3.
    • Ação: Criar Evento na agenda do Proprietário.
    • Título: "Ensaio Pré-Pago [Banda do cliente]".
    • Participantes (Attendees): Adicione o e-mail do cliente (que veio junto nos metadados do Webhook da InfinitePay ou foi enviado do Next.js para banco temporário). Isso garante o disparo oficial do Google para o cliente.

Fase 4: Implantação & Passagem de Bastão

Passo 4.1

Deployment Vercel (Next.js)

  1. Suba o projeto Next.js na Vercel conectado ao seu repositório GitHub.
  2. Preencha a aba de Environment Variables religiosamente: 
    GOOGLE_CLIENT_ID=xxx
GOOGLE_CLIENT_SECRET=xxx
NEXTAUTH_URL=https://site-do-estudio.com.br
NEXTAUTH_SECRET=umahashbemsegura
INFINITEPAY_API_KEY=xxx
N8N_WEBHOOK_URL=https://seu-n8n.com/webhook/xxx
Passo 4.2

Validação do Proprietário

  1. Agende uma reunião de Handover com o dono do estúdio.
  2. Mostre que agora ele não precisa olhar painel de Quaddro. A notificação de agendamento acende direto no Google Agenda, e o dinheiro cai direto no saldo da conta do banco da InfinitePay/Mercado Pago sem taxas de intermediação.
Vantagem Estratégica do Lobo Solitário: Este sistema não possui banco de dados relacional para gerenciar, nem disparo customizado de WhatsApp para pagar contas na Z-API. As engrenagens são do próprio Google. Isso é uma máquina de vendas "Set and Forget".
